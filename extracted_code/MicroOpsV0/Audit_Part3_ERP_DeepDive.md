# Phase 3: ERP Architecture & Business Logic Stress Test

Perspective: Senior ERP Architect & Security Consultant. Scope: extracted_code/MicroOpsV0 frontend SPA + Node/Express/Postgres backend. No source files modified; findings only.
Severity legend: Critical / High / Medium (business risk).

## Module: Auth / RBAC
- Severity: Critical — Frontend auth is PIN-only (plain text in App.Data), no hashing or server verification; backend expects JWT. Remote API calls send empty Authorization header -> either fail (availability) or tempt disabling auth (total exposure).
- Severity: High — Role taxonomies diverge (frontend admin/sales/warehouse/production/user vs backend admin/manager/sales/warehouse/readonly). Mismatch can deny legitimate users or overgrant access.
- Severity: High — UI hides routes via sidebar but does not enforce RBAC in views; direct hash (#settings, #documents) renders and mutates App.Data regardless of role.
- Severity: High — Server RBAC gaps: /api/config GET requires only authenticate; /api/orders, /api/documents, /api/inventory products/components are auth-only exposing financial/stock to any logged-in user.
- Severity: High — Lock/unlock reuses PIN with no throttle; brute force possible while session is active.
- Severity: Medium — Sessions table cleaned only on logout; no expiry purge. Token hash is suffix only; revocation is coarse (delete by user_id).
- Severity: Medium — Static hosting of project root can leak .env.example and docs; helmet CSP not configured; inline scripts abundant.

## Module: Data Integrity & Relationship Logic
- Severity: Critical — Frontend App.Data is primary data source; App.Api set to remote but UI reads local state. Server updates are invisible -> decisions on stale data.
- Severity: Critical — No sync/conflict detection between IndexedDB/localStorage and Postgres; last writer wins across tabs, creating silent divergence.
- Severity: High — Orphan risk: deleting customers blocked only by orders count; documents and movements still reference deleted customer/order without cascade.
- Severity: High — inventory_movements.item_id has no FK; movements can point to nonexistent products/components; stock ledger cannot be trusted.
- Severity: High — Order ship/confirm uses current_stock check then update; concurrent transactions can oversell without SERIALIZABLE + row locks per line.
- Severity: High — Document creation copies totals from order without recompute; discount/VAT drift yields financial discrepancies.
- Severity: High — No soft delete for orders/documents/products/components; deletes are hard and partial (some routes block by status), leaving dangling references.
- Severity: Medium — BOM table unused; production orders do not consume components; manufactured stock valuation incorrect.
- Severity: Medium — Sequences not year-rotated; numbering can duplicate across years, violating legal/invoice compliance.

## Module: Inventory / Movements / Batches
- Severity: Critical — Field mismatch: UI uses stock/minStock; backend uses current_stock/min_stock. Remote mode will misreport levels and mis-trigger alerts.
- Severity: High — Adjust endpoint allows absolute set with movement_type=adjustment; no approvals or dual control. Any authorized user can zero warehouse.
- Severity: High — No reservation mechanism; confirmed orders do not reserve stock. Two shipments can consume same quantity.
- Severity: High — No link to batches/expiry; expired lots can be shipped; batch page is UI-only.
- Severity: Medium — Alerts endpoint uses DB min_stock only; UI uses reorderPoint; inconsistent replenishment signals.
- Severity: Medium — Movements lack reference FK to orders/shipments; traceability weak; audits cannot join reliably.

## Module: Sales Orders / Documents / Billing
- Severity: Critical — Posting invoice not tied to shipment; revenue can be recognized before goods leave, causing financial misstatements.
- Severity: Critical — Mark-paid records no amount/date; partial/over payments undetectable; AR aging unreliable.
- Severity: High — Document delete allowed only in draft; no trash/restore like UI suggests. Hard deletes drop audit trail context.
- Severity: High — Multi-currency unsupported; totals assume EUR. Cross-currency deals will be wrong.
- Severity: Medium — Line-level rounding policy undefined (qty 4dp vs totals 2dp); cumulative rounding drift on large invoices.

## Module: Pricing / Products / Components
- Severity: High — Price lists exist only in client state; backend has no schema. Negotiated prices lost when API enforced.
- Severity: High — Cost/margin data exposed in client to all roles (dealerPrice, avgPurchasePrice). Warehouse users can see margins.
- Severity: Medium — Product identifiers differ (internalArticleNumber vs sku); risk of duplicates and broken joins across systems.

## Module: Tasks / Automation
- Severity: High — Batch-expiry task generation runs client-side only; multi-user environments miss tasks unless someone opens UI.
- Severity: Medium — Automation writes to App.Data without locking; duplicate tasks possible when multiple tabs active.

## Module: Configuration / Backup / Compliance
- Severity: Critical — Backup create/restore executes pg_dump/psql synchronously via exec; large DB blocks event loop, potential downtime. Restore runs arbitrary .sql filename from body (path join but no whitelist).
- Severity: High — Backup dir default /tmp; may be world-readable. Checksums not encrypted; download requires admin but static hosting could leak.
- Severity: High — Config GET is auth-only; leaks bank, VAT, IBAN to all logged-in users.
- Severity: Medium — No retention for backup_log; unbounded growth; no scheduled verification.

## Module: UI Rendering / Performance
- Severity: Critical — List pages render entire dataset without pagination/virtualization (inventory, orders, documents, dashboard). 10k rows will freeze browser.
- Severity: High — Dashboard recomputes aggregates via multiple filters/reduces on each render; O(n*m) heavy load at scale.
- Severity: High — App.DB.save serializes full App.Data on many UI toggles; O(n) per click with toast spam; blocking at large n.
- Severity: Medium — Sorting/filtering done client-side even in remote mode; no LIMIT/OFFSET, wasting network and CPU.

## Module: Dashboard / Health / Monitoring
- Severity: High — Dashboard KPIs use local App.Data; server /api/config dashboard stats differ. Execs will see conflicting numbers.
- Severity: Medium — Health check relies on navigator.storage; may falsely report healthy; no server dependency checks surfaced in UI.

## Module: Race Conditions & Concurrency
- Severity: Critical — Order confirm/ship not SERIALIZABLE; concurrent confirmations can oversell stock.
- Severity: High — Local data + multiple tabs: DB.save overwrites whole state; no versioning or locks.
- Severity: Medium — SessionManager vs Services.SessionManager timers may conflict; double logout could interrupt saves.

## Module: Performance Hotspots (Specific)
- Severity: High — inventory.js builds big HTML via map/join each tab; no pagination; will hang on large inventories.
- Severity: High — dashboard.js multiple reduce/filter passes; lacks memoization or server offload.
- Severity: Medium — App.Api list sorts client-side after fetching all data; should delegate to DB.

## Module: Frontend Exposure
- Severity: High — App.Data shipped to client includes IBAN/BIC/bankName, pricing, and config for all roles; any user can extract via devtools.
- Severity: Medium — Audit log array (if populated) visible to any role; could leak security events.
